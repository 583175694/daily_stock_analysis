# AI 选股 PRD

## 1. 文档定位

本文档是 AI 选股功能的产品需求文档，回答的是以下问题：

- 这个功能给谁用
- 解决什么问题
- 要做哪些功能
- 先做什么，后做什么
- 怎样判断这一版做成了

对应关系：

- `PRD` 解决“做什么、为什么做、先后顺序是什么”
- `设计方案` 解决“怎么分层、怎么落地、怎么减少和上游冲突”

相关文档：

- `docs/ai-stock-picker-design.md`

## 2. 产品背景

当前项目已经能很好地完成以下任务：

- 对用户指定的股票做单股分析
- 做大盘复盘
- 提供 Agent 问股与深度研究
- 进行历史回测
- 管理持仓和组合风险

但用户仍需要自己先决定“分析哪些股票”。这意味着当前系统更像“分析器”，还不是“机会发现系统”。

AI 选股功能的目标，是把项目从“收到股票后做分析”，升级为“先从股票池里发现候选，再给出解释和后续动作建议”。

## 3. 问题定义

当前存在的核心问题：

- 用户每天要先自己挑股票，分析入口仍然依赖人工前置判断
- 自选股一多以后，很难快速识别真正值得关注的少数标的
- 单股分析很强，但缺少“批量筛选 + 排序 + 解释”的上游步骤
- 现有回测能验证分析结果，但还不能系统验证“选出来的候选是否真的有效”
- 缺少从“市场环境 -> 候选股 -> 深度分析 -> 后验跟踪”的完整闭环

## 4. 产品目标

### 4.1 核心目标

- 让系统具备从股票池中自动筛选候选股的能力
- 让候选结果可解释，而不是只给黑盒排序
- 让候选结果可跟踪、可回看、可验证
- 让选股能力与现有单股分析、Agent、通知、回测、组合模块形成闭环

### 4.2 业务目标

- 提升用户每日使用频率
- 提升从“跑分析”到“采取行动”的转化效率
- 提升批量场景下的决策效率
- 为后续更高级的策略模板、个性化权重、组合联动打基础

### 4.3 非目标

- 第一阶段不追求全自动实盘交易
- 第一阶段不做复杂量化回测框架重构
- 第一阶段不做因子研究平台
- 第一阶段不做社交型选股社区功能

## 5. 目标用户

### 5.1 核心用户

- 已经在使用本项目做日常股票分析的活跃用户
- 拥有稳定自选股池，需要每天筛选重点关注标的的用户
- 同时关注 A 股、港股、美股中的至少一个市场的用户

### 5.2 典型用户类型

#### 用户类型 A：短线交易者

特点：

- 更关注趋势、量能、情绪、板块轮动
- 更需要每天快速筛出强势候选
- 对“为什么入选”和“什么时候观察/止损”敏感

#### 用户类型 B：波段交易者

特点：

- 更关注趋势启动、回调承接、景气变化
- 更需要候选排序和风险解释
- 希望结合市场环境判断是否值得跟踪

#### 用户类型 C：组合管理者

特点：

- 已有真实持仓
- 更关注新候选和现有仓位是否重复暴露
- 希望候选结果能联动组合风险视角

## 6. 核心价值主张

AI 选股不是替代现有单股分析，而是在其前面新增一个“发现机会”的入口层。

用户最终获得的价值是：

- 系统先帮我找机会
- 系统告诉我为什么这些票值得看
- 系统告诉我哪些票不该看
- 系统能持续验证自己的选股效果

## 7. 核心使用场景

### 场景 1：每日盘后选股

用户在盘后运行一个选股任务，系统从指定股票池中筛出 Top N 候选，并给出入选理由、风险点和下一步建议。

### 场景 2：从候选到单股深挖

用户看到候选结果后，对其中 1 到 3 只股票发起单股详细分析或 Agent 追问，继续深挖。

### 场景 3：从候选到观察池

用户把候选结果中暂时不买、但值得继续跟踪的股票加入观察池或自选股列表。

### 场景 4：验证最近 30 天哪种模板更有效

用户查看过去一段时间不同选股模板的命中率、收益率和回撤表现，决定是否继续使用某个模板。

### 场景 5：结合持仓做增量决策

用户已有持仓，系统在输出候选时同时提示行业暴露、重仓重合或风险重复问题。

## 8. 产品原则

- 先筛选，再解释，不做纯黑盒推荐
- 先做可回测、可追踪的能力，再做更复杂的 AI 增强
- 先内置模板，后开放外部策略输入
- 先服务已有用户的日常流程，再考虑更宽泛的外部用户
- 先保证结果可理解，再追求结果“看起来很智能”
- AI 负责解释和复核，不应完全替代结构化规则和结构化评分

## 9. 功能范围总览

AI 选股功能分为三层能力：

- 候选生成
- 候选解释
- 候选验证

对应的产品能力为：

- 股票池管理
- 选股模板
- 手动和定时运行
- 结果列表与详情
- 候选解释
- 一键后续动作
- 历史记录与跟踪
- 效果统计
- 组合联动

## 10. 版本规划

### V1：MVP 版本

目标：

- 做出第一版可用、可解释、可保存结果的 AI 选股能力

功能范围：

- 股票池选择
- 选股模板选择
- 手动运行一次选股
- 候选列表展示
- 候选详情展示
- 一键发起单股分析
- 一键加入自选股
- 历史任务与结果回看

不包含：

- 定时自动执行
- 结果后验跟踪看板
- 组合风险联动
- 模板效果对比面板

### V2：验证增强版本

目标：

- 让选股结果具备后验验证和复盘能力

功能范围：

- 参数化模板
- 候选股后续表现跟踪
- 选股模板效果统计
- 胜率、收益率、回撤等指标展示
- 选股结果摘要通知

### V3：闭环增强版本

目标：

- 让 AI 选股真正融入用户日常工作流

功能范围：

- 自然语言策略输入
- 定时自动选股
- 盘前/盘后任务计划
- 组合暴露联动
- 按市场环境动态调整模板
- 候选与持仓的联动建议

### 10.1 策略输入模式演进

AI 选股的策略来源分三层：

#### 第一层：内置模板

定义：

- 系统预先定义模板
- 用户只负责选择，不负责定义模板本身

适用版本：

- V1

价值：

- 结果稳定
- 易于解释
- 易于回测
- 实现成本最低

#### 第二层：参数化模板

定义：

- 用户基于内置模板做受控调参
- 调整的是模板参数，不是模板结构本身

适用版本：

- V2

典型参数：

- 候选数量
- 风险偏好
- 市场范围
- 新闻权重

#### 第三层：自然语言策略输入

定义：

- 用户输入一句策略描述
- AI 先将其解释成结构化模板或模板参数
- 系统执行的仍是归一化后的结构化策略

适用版本：

- V3

关键约束：

- 自然语言不能直接绕过模板体系
- 用户需要知道系统最终按什么逻辑执行筛选

### 10.2 当前产品结论

V1 明确结论：

- 只支持内置模板
- 不支持自由输入策略直接执行
- 不支持让 AI 自由决定筛选规则

原因：

- V1 首要目标是稳定、可解释、可回看
- 如果过早开放自由输入，前后端契约、回测口径和异常兜底都会显著复杂化

## 11. MVP 功能需求

### 11.1 股票池选择

用户能力：

- 选择一个用于筛选的股票池

MVP 范围：

- 使用当前自选股列表
- 使用预定义股票池
- 允许后续扩展到指数成分和板块池

验收标准：

- 用户可以明确知道本次选股扫描的是哪一组股票
- 结果页能展示股票池名称和扫描数量

### 11.2 选股模板选择

用户能力：

- 按不同风格运行选股

MVP 推荐模板：

- 趋势突破
- 强势回调
- 板块龙头
- 事件催化
- 均衡综合

MVP 约束：

- 模板由系统内置
- 用户只能选择模板，不能自由输入自然语言策略
- 模板排序规则由结构化评分驱动，AI 不直接决定排名

验收标准：

- 每个模板都有清晰名称和简短说明
- 同一股票池可以切换不同模板运行

### 11.3 手动运行选股

用户能力：

- 在 Web 页面主动发起一次选股任务

系统行为：

- 接收股票池和模板参数
- 创建任务
- 返回执行中状态
- 最终生成候选结果

验收标准：

- 用户可看到任务状态
- 任务完成后可直接进入结果页
- 异常时有明确失败提示

### 11.4 候选结果列表

用户能力：

- 查看本次选股的 Top N 候选

MVP 列表字段建议：

- 排名
- 股票代码
- 股票名称
- 总分
- 主标签
- 简短入选理由
- 风险提示摘要

验收标准：

- 用户能快速理解“为什么这只股票排在前面”
- 默认结果数量可控，列表信息密度适合快速浏览

### 11.5 候选详情

用户能力：

- 查看单个候选股的详细信息

MVP 详情建议包含：

- 总分与各维度分
- 入选原因
- 核心风险
- 关键观察点
- 板块或主题上下文
- 可选的最近相关新闻摘要

验收标准：

- 候选详情不能只给结论，必须给原因
- 用户能从详情页做下一步动作

### 11.6 一键后续动作

用户能力：

- 从候选结果继续进入后续流程

MVP 动作包括：

- 发起单股详细分析
- 打开 Agent 问股继续追问
- 加入自选股

验收标准：

- 用户不需要手动复制股票代码再去别的页面操作

### 11.7 历史记录

用户能力：

- 回看历史选股任务和历史候选结果

MVP 要求：

- 可查看最近任务
- 可查看某次任务的结果快照
- 可按时间回看

验收标准：

- 历史结果不会因为实时数据变化而完全丢失当时上下文

## 12. V2 功能需求

### 12.0 参数化模板

用户能力：

- 在已有模板基础上调整少量可控参数

建议参数：

- 候选数量
- 风险偏好
- 市场范围
- 新闻权重

验收标准：

- 用户可以在不重新定义模板逻辑的前提下做轻量个性化调整
- 参数变化对结果有清晰、可预期的影响

### 12.1 结果后验跟踪

用户能力：

- 查看候选股后续表现

建议指标：

- T+1 表现
- 3 日表现
- 5 日表现
- 10 日表现
- 最大回撤

验收标准：

- 用户能清楚知道“选出来之后表现怎样”

### 12.2 模板效果统计

用户能力：

- 对比不同模板在一段时间内的表现

建议指标：

- 入选数量
- 胜率
- 平均收益
- 中位收益
- 最大回撤

验收标准：

- 用户能根据结果决定是否继续使用某模板

### 12.3 通知摘要

用户能力：

- 选股结束后收到摘要通知

通知内容建议：

- 股票池
- 模板
- Top N 候选
- 每只候选的一句话原因

验收标准：

- 不打开 Web 也能快速知道本次有哪些重点候选

## 13. V3 功能需求

### 13.1 定时自动选股

用户能力：

- 设定盘前或盘后自动运行选股

验收标准：

- 用户无需手动打开页面也能稳定拿到每日候选

### 13.2 组合联动

用户能力：

- 从组合视角审视候选结果

建议输出：

- 是否与现有重仓同板块
- 是否增加行业集中度
- 是否加剧单市场暴露
- 是否适合作为观察标的而非立即加仓

验收标准：

- 候选结果不只是“好股票列表”，而是“适合当前账户的候选列表”

### 13.3 市场环境联动

用户能力：

- 让选股结果与市场环境一致

建议能力：

- 根据市场强弱动态调整模板权重
- 在弱市减少激进候选
- 在强市提升趋势候选权重

验收标准：

- 结果风格能反映市场环境，而不是始终输出一套固定口径

### 13.4 自然语言策略输入

用户能力：

- 通过一句自然语言描述发起选股

系统要求：

- 先把自然语言解释成结构化模板
- 用户可查看或确认解释后的策略摘要
- 系统再执行选股

验收标准：

- 自然语言输入不会直接绕过结构化模板体系
- 用户能知道系统最终按什么逻辑执行了筛选

## 14. 页面需求

建议新增独立页面：

- `AI 选股`

页面模块建议：

- 顶部参数区
- 任务状态区
- 候选结果列表
- 候选详情抽屉或详情区
- 历史任务入口

关键要求：

- 结果列表适合快速扫盘
- 详情适合做判断
- 和现有单股分析、问股、历史页面可顺畅跳转

## 15. 输出物定义

一次 AI 选股任务的标准输出应包含：

- 任务元信息
- 股票池信息
- 模板信息
- 候选结果列表
- 每只候选的评分摘要
- 每只候选的解释摘要
- 任务生成时间

后续增强输出：

- 候选后验表现
- 模板整体表现
- 组合联动提示

## 16. 关键用户流程

### 流程 A：手动选股

- 用户进入 AI 选股页
- 选择股票池
- 选择模板
- 点击运行
- 查看任务执行状态
- 查看候选结果
- 对重点候选发起单股分析或问股

### 流程 B：结果回看

- 用户进入历史任务
- 打开某次选股结果
- 查看当时的候选列表和解释
- 对比后续表现

### 流程 C：加入观察池

- 用户查看候选结果
- 勾选其中部分股票
- 一键加入自选股或观察池

## 17. 核心指标

### 17.1 产品使用指标

- AI 选股页周活用户数
- 人均每周运行任务次数
- 从选股结果进入单股分析的转化率
- 从选股结果加入自选股的转化率

### 17.2 结果质量指标

- Top N 候选的 1 日、3 日、5 日表现
- 模板间胜率差异
- 用户手动保留候选的比例
- 用户二次进入详细分析的比例

### 17.3 系统效率指标

- 单次选股任务耗时
- 候选结果生成成功率
- 单次任务调用 LLM 成本

## 18. 非功能要求

- 结果必须可解释，不能只给分数
- 任务失败时要给明确错误反馈
- 对外部数据源波动应尽量 fail-open
- 任务耗时应可控，不能明显拖垮日常使用体验
- 历史结果需要可回看，避免实时数据完全覆盖当时状态

## 19. 验收标准

### V1 验收标准

- 用户能从页面手动运行一次选股
- 系统能输出 Top N 候选列表
- 每只候选有明确入选理由和风险说明
- 用户能从候选结果一键发起单股分析
- 历史任务可以回看

### V2 验收标准

- 系统能自动跟踪候选结果后续表现
- 用户能看到模板效果统计
- 任务完成后可以输出摘要通知

### V3 验收标准

- 用户能配置定时自动选股
- 候选结果能体现组合和市场环境约束
- 用户能把选股纳入日常固定流程

## 20. 暂不纳入本轮范围的能力

- 自动下单或交易执行
- 多用户协作与权限系统
- 社区化策略分享
- 因子研究工作台
- 高自由度可视化拖拽策略编辑器
- V1 中的自然语言自由策略输入

## 21. 当前建议

建议先以 V1 为目标，优先做出一个“可运行、可解释、可回看”的 MVP。

V1 做成之后，再进入 V2 和 V3。原因是：

- 先验证用户是否真的会使用这个入口
- 先验证候选结果是否具备基本质量
- 先验证模板和股票池设计是否合理

只有 V1 的使用和结果都成立，后续的跟踪、组合联动、定时自动化才有继续投入的价值。
