# AI 选股扩展方案设计

## 1. 背景

本项目已经具备较完整的股票分析底座：

- 多市场行情与信息源接入
- 单股分析、大盘复盘、Agent 问股、回测、组合管理
- Web、桌面端、Bot、API、通知、定时任务等使用面

当前最自然的能力延伸，不是继续强化“分析一只已经给出的股票”，而是增加“先从股票池里筛出值得关注的候选，再进行解释、跟踪和验证”的 AI 选股能力。

但本项目是持续演进的开源仓库，二次开发时除了功能本身，还必须兼顾以下现实目标：

- 仍然能够较低成本跟进上游更新
- 尽量减少和上游主干代码的冲突面
- 不把 AI 选股功能做成难以维护的深耦合补丁
- 在同一个仓库内完成分层，不外化成独立服务

本文档用于给出 AI 选股扩展的多方案设计，并明确推荐方案与落地边界。

## 2. 设计目标

- 在同仓库内新增一个独立的 AI 选股业务域
- 最大化复用现有数据、分析、通知、回测、Web 与 Agent 能力
- 最小化对现有单股分析主流程和高频变动文件的侵入
- 让 AI 选股模块从第一天起就具备清晰分层和可迁移性
- 支持从 MVP 到完整选股系统的渐进式演进

## 3. 非目标

- 不在第一阶段把 AI 选股做成仓外独立服务
- 不重写现有单股分析、历史记录、回测或组合管理主链路
- 不把选股结果硬塞进现有分析历史表的既有语义
- 不为了“统一体验”大范围改造首页、设置页、主调度入口

## 4. 关键约束

- 代码仍在同一仓库内开发
- 优先做加法式扩展，而不是改写现有模块
- 上游后续仍可能持续变更 API、Web、Agent、配置、数据源与调度逻辑
- 新能力后续可能继续长大，因此现在的边界设计必须考虑未来拆分成本

## 5. 设计原则

- 业务域独立：AI 选股是新业务域，不是现有单股分析流程的附属步骤
- 薄集成：与现有系统只在少量入口和适配层交汇
- 分层复用：复用现有公共能力，但不直接散落依赖主仓内部细节
- 数据隔离：新表、新任务、新配置命名空间，避免污染现有语义
- 渐进演进：先实现可验证的 MVP，再叠加 AI 解释、跟踪、组合联动与调度
- 先内置后开放：先固化稳定模板，再开放参数化和自然语言输入
- 冲突集中：把未来和上游同步时的冲突，尽量集中在少数挂载点

## 6. 方案对比

### 方案 A：深度集成到现有单股分析主流程

做法：

- 在现有分析流水线前增加“全市场筛选”
- 复用现有单股分析对象、报告结构、任务系统与历史表
- 通过现有 Web 首页和调度入口直接承载选股能力

优点：

- 开发初期最快
- 复用路径最直接
- 用户入口最统一

缺点：

- 容易侵入现有核心分析主流程
- 与上游持续演进的冲突面最大
- 选股和单股分析职责容易混淆
- 后期做独立排名、跟踪、验证时会越来越难

适用场景：

- 只想快速验证一个很轻量的内部实验

判断：

- 不推荐作为正式方案

### 方案 B：同仓独立业务域 + 薄集成挂载

做法：

- 在 `src/` 下新增 AI 选股专属目录
- 选股核心逻辑、仓储、任务、结果跟踪都独立实现
- 通过少量 API 路由、前端页面、通知入口、调度入口与现有系统集成
- 复用现有数据源、Agent、回测、通知、组合能力，但通过适配层接入

优点：

- 同仓复用效率高
- 冲突面可控
- 业务边界清晰
- 后续做深不会立刻拖垮主干结构
- 如果未来要抽离，迁移成本也较低

缺点：

- 初期比方案 A 多一层设计与封装成本
- 需要克制“顺手改主流程”的冲动
- 需要维护自己的数据表、任务和配置边界

适用场景：

- 当前项目阶段的主推荐方案

判断：

- 推荐

### 方案 C：同仓插件化扩展层

做法：

- 先在仓库内建立一个通用扩展平台
- AI 选股作为首个插件接入
- 后续再把更多能力做成插件

优点：

- 长期扩展性理论上更强
- 边界会更规范

缺点：

- 现在就引入插件框架，工程复杂度偏高
- 容易为了抽象而抽象
- 当前只有一个新域时，性价比不高

适用场景：

- 未来明确还会有多个大型业务域并行扩展

判断：

- 可作为长期演进方向，不建议作为第一阶段落地方式

### 方案 D：仓外独立服务

做法：

- AI 选股放到单独服务或单独仓库
- 通过 API 与当前项目通信

优点：

- 与上游主仓冲突最小
- 技术边界最清晰

缺点：

- 当前阶段集成成本高
- 需要重复建设运行、配置、部署、权限与通知链路
- 不符合本次“同仓开发”的决策前提

判断：

- 本轮不采用，仅作为对照方案保留

## 7. 推荐结论

推荐采用：

- 方案 B：同仓独立业务域 + 薄集成挂载

不建议：

- 把选股逻辑直接揉进现有单股分析主流程
- 一开始就做完整插件平台
- 当前阶段外化为独立服务

推荐理由：

- 兼顾开发效率与后续可维护性
- 能最大限度复用现有底座
- 能把未来与上游同步时的冲突控制在较小范围
- 能为后续能力继续变复杂预留空间

## 8. 推荐架构

### 8.1 总体结构

推荐在 `src/` 下新增独立业务域目录，例如：

```text
src/stock_picker/
  domain/
  application/
  infrastructure/
  interfaces/
```

推荐职责划分：

- `domain/`
  - 股票池定义
  - 初筛规则
  - 因子与评分模型
  - 候选股实体
  - 排名与入选逻辑
  - 跟踪结果与验证规则
- `application/`
  - 运行一次选股任务
  - 根据股票池生成候选
  - 调用 AI 做解释与复核
  - 保存结果、生成摘要、触发通知
  - 跟踪历史选股表现
- `infrastructure/`
  - 现有数据源适配
  - 现有分析能力适配
  - Agent/LLM 适配
  - 通知适配
  - 数据库存储实现
  - 回测与组合能力适配
- `interfaces/`
  - API 调用入口
  - 定时任务入口
  - 可选 CLI 入口

### 8.2 与现有仓库的推荐薄集成点

建议只在少数位置做挂载：

- 新增 API 路由注册
- 新增 Web 页面路由和侧边栏入口
- 新增单独的选股运行入口
- 可选地增加一个定时任务挂点

建议保持为薄改动的现有区域：

- `api/v1/router.py`
- Web 路由入口
- 侧边栏导航
- 少量调度挂载文件

## 9. 不建议触碰的区域

以下区域是高冲突风险区，除非有明确必要，不建议作为第一阶段主改动面：

- 现有单股分析核心流水线
- 现有分析报告主结构与历史记录主语义
- 数据源总入口和 fallback 主逻辑
- 全局配置主模型的大规模字段扩张
- 首页主看板的大面积重构
- 设置页的大规模侵入式改造
- `main.py` 中持续增长的既有启动参数和主调度分支

原因：

- 这些区域是上游高频变化区
- 一旦深度侵入，后续同步上游时冲突会快速放大

## 10. 模块边界设计

### 10.1 股票池层

建议把“股票池”作为独立能力，而不是简单复用 `STOCK_LIST`：

- 全市场池
- 指数成分池
- 行业板块池
- 自定义观察池
- 当前持仓关联池

建议第一阶段先支持：

- 手工定义股票池
- 从现有自选股派生股票池
- 从板块或指数成分生成候选池

### 10.2 初筛层

这一层只做确定性规则，不做 LLM 推理。

建议承载：

- 市场与标的基础过滤
- 流动性过滤
- 价格与成交额过滤
- 趋势条件过滤
- 风险条件过滤
- 数据完整性过滤

目标：

- 尽快把全量股票池缩小到一个可解释、可计算、可回测的候选集合

### 10.3 评分层

建议采用结构化打分，而不是让 LLM 直接决定选股结果。

推荐评分维度：

- 技术趋势分
- 量价结构分
- 板块强度分
- 新闻催化分
- 基本面质量分
- 风险扣分
- 市场环境分
- 组合重合度扣分

建议结果形态：

- 总分
- 各维度子分
- 主要加分项
- 主要减分项

### 10.4 AI 复核与解释层

AI 层建议只处理“复核、解释、归纳”，不直接替代基础筛选和基础打分。

建议 AI 只面对：

- 初筛后排名前 N 的股票
- 已经准备好的结构化特征与情报摘要

建议 AI 产出：

- 入选理由
- 核心风险
- 关注催化
- 建议观察点
- 是否值得进入重点分析列表

### 10.5 跟踪与验证层

AI 选股要长期可用，必须从一开始就设计结果跟踪。

建议内置：

- T+1、3 日、5 日、10 日表现跟踪
- 胜率、收益率、回撤统计
- 各策略模板的效果对比
- AI 解释与后续表现的对应分析

### 10.6 策略来源设计

AI 选股的策略来源不建议一步到位开放，而应分三层设计。

#### 层 1：内置模板

特点：

- 模板由系统定义
- 过滤条件、评分权重和模板说明都稳定可控
- 用户只做选择，不直接定义规则

适用阶段：

- V1

价值：

- 最利于可解释性
- 最利于回测和横向比较
- 最利于前后端契约稳定

#### 层 2：参数化模板

特点：

- 用户在既有模板上调整少量参数
- 调整对象是模板参数，而不是模板结构本身

适用阶段：

- V2

示例：

- 候选数量
- 风险偏好
- 新闻权重
- 市场范围

价值：

- 兼顾灵活性和结构化约束
- 不会让策略输入彻底失控

#### 层 3：自然语言策略输入

特点：

- 用户输入一句策略描述
- AI 将其解析为结构化模板或参数化模板
- 系统执行的是“归一化后的结构化策略”，不是原始自然语言

适用阶段：

- V3

关键约束：

- 自然语言输入不能直接跳过模板层
- 必须保留归一化结果，便于结果解释和后续回看

#### 当前架构结论

推荐架构从第一天就预留三种对象，但只实现第一种：

- `BuiltinTemplate`
- `TemplateOverrides`
- `StrategyIntent`

其中：

- V1 只启用 `BuiltinTemplate`
- V2 启用 `TemplateOverrides`
- V3 才引入 `StrategyIntent -> BuiltinTemplate/TemplateOverrides` 的归一化流程

#### 为什么不建议 V1 直接开放外部输入

如果在 V1 直接允许自由输入策略，会带来以下问题：

- 无法稳定回测
- 难以在前端展示真实执行规则
- 错误输入和模糊输入难以兜底
- 结果稳定性和复现性差
- 和模板效果统计难以统一口径

因此，V1 更合理的路线是：

- 先让 AI 在模板结果上做解释
- 不让 AI 在 V1 中自由定义模板

## 11. 数据存储设计

建议新增独立表，而不是改写现有分析历史表。

推荐最小表集合：

- `picker_tasks`
  - 一次选股任务
- `picker_candidates`
  - 某次任务的候选股结果
- `picker_candidate_scores`
  - 候选股各维度评分明细
- `picker_tracking`
  - 候选股后验表现跟踪
- `picker_universes`
  - 自定义股票池定义
- `picker_presets`
  - 选股策略模板配置

设计原则：

- 表名前缀统一
- 全部采用新增表，不修改旧表语义
- 允许未来增加字段，但避免和现有表产生强耦合

## 12. 配置设计

为了减少和上游配置体系的冲突，不建议第一阶段就把大量选股配置塞进全局配置中心。

推荐分两步：

### 第一阶段

- 使用独立配置文件，例如 `config/stock_picker.yaml`
- 主系统只通过一个轻量入口知道该配置文件路径
- 选股域内部自行解析模板、规则、评分权重与股票池定义

### 第二阶段

- 对少量高频、用户强感知的配置再逐步纳入主设置页
- 只把真正稳定且常用的配置并入全局 Schema

这样做的价值：

- 初期实现快
- 全局配置冲突少
- 后续功能稳定后再收编，不会把试验性字段过早固定化

## 13. API 设计建议

建议新增独立命名空间，例如：

- `/api/v1/picker/run`
- `/api/v1/picker/tasks`
- `/api/v1/picker/tasks/{task_id}`
- `/api/v1/picker/results/{task_id}`
- `/api/v1/picker/tracking`
- `/api/v1/picker/universes`
- `/api/v1/picker/presets`

建议 API 职责：

- 运行选股
- 查询任务和结果
- 查看候选股评分明细
- 查看后验跟踪结果
- 管理股票池
- 管理策略模板

不建议：

- 复用现有分析任务 API 直接承载选股任务

原因：

- 选股任务的粒度、状态与产出物不同于单股分析
- 如果强行复用，会把任务语义搅乱

## 14. Web 设计建议

建议新增独立页面，而不是挤进首页。

推荐页面：

- `AI 选股页`

建议承载能力：

- 选择股票池
- 选择选股模板
- 手动运行选股
- 查看任务状态
- 查看 Top N 候选
- 查看候选股评分详情
- 查看 AI 解释
- 一键发起单股详细分析
- 一键加入自选股或观察池

建议只做以下薄改动：

- 新增页面路由
- 新增导航入口

不建议：

- 在首页主面板中直接承载完整选股工作流

## 15. 调度与运行方式设计

为减少对现有 `main.py` 的侵入，建议分阶段处理。

### 第一阶段

- 先通过 API 和 Web 页面手动运行
- 如需命令行，新增单独入口脚本，例如 `scripts/run_picker.py`

### 第二阶段

- 再考虑接入现有定时调度体系
- 只新增一个轻量挂点，让调度器在指定时机调用选股应用服务

这样做的价值：

- 不必一开始就在 `main.py` 中新增大量参数与分支
- 与上游调度相关改动的冲突更少

## 16. 与现有能力的复用关系

建议复用但要通过适配层接入：

- 数据源与行情获取
- 新闻与搜索能力
- 基本面聚合能力
- Agent/LLM 解释能力
- 通知发送能力
- 组合风险能力
- 回测与结果评估能力

不建议的复用方式：

- 在新模块里直接到处调用旧模块内部函数
- 依赖旧模块的隐式上下文和历史副作用

推荐方式：

- 在 `infrastructure/` 中做少量稳定适配器
- 让 `application/` 和 `domain/` 只看到选股语义，而不是直接看到主仓杂乱的内部细节

## 17. 开发阶段建议

### M0：边界与骨架

- 建立 `src/stock_picker/` 目录骨架
- 建立独立仓储与独立数据表
- 新增最小 API 和最小页面占位

目标：

- 先把边界立住

### M1：确定性 MVP

- 股票池
- 初筛规则
- 多维评分
- Top N 排名
- 结果持久化

目标：

- 做出可运行、可解释、可回测的基础选股系统

### M2：AI 解释增强

- 对前 N 候选股做 AI 复核
- 生成入选理由、风险点、催化点
- 页面支持候选详情查看

目标：

- 从“筛选器”升级到“可解释的 AI 选股器”

### M3：后验跟踪

- 自动记录候选股后续表现
- 输出模板胜率、收益率、回撤
- 支持按周期查看策略效果

目标：

- 让系统具备真实迭代能力

### M4：组合联动

- 和持仓、行业暴露、风险视角打通
- 给出“适合加入观察池”还是“与现有仓位重复”的建议

目标：

- 让选股结果和个人持仓决策产生闭环

### M5：定时任务与通知

- 接入调度
- 盘前或盘后自动跑选股
- 自动发送 Top N 候选摘要

目标：

- 从手动使用走向每日自动化

## 18. 冲突最小化策略

为了保留持续拉取上游更新的能力，建议在开发过程中坚持以下约束：

- 能新增文件就不要修改旧文件
- 能在独立目录实现就不要把逻辑散落到 `src/services/`、`src/repositories/` 各处
- 能独立建表就不要复用旧表语义
- 能新开页面就不要大改首页
- 能独立脚本运行就不要先侵入 `main.py`
- 能通过适配器复用就不要直接耦合旧模块内部细节

推荐把未来冲突集中在少数文件：

- API 路由注册文件
- 前端路由文件
- 导航入口文件
- 少量调度挂载文件

## 19. 风险与应对

风险一：

- 业务边界一开始设计清楚，但实现时逐渐侵入主流程

应对：

- 对“是否修改高风险区域”设置明确评审门槛

风险二：

- 评分模型过早依赖 LLM，导致结果不可回测、成本过高

应对：

- 先做规则筛选和结构化评分，AI 只做复核与解释

风险三：

- 过早把大量实验性字段塞进全局配置体系

应对：

- 先独立配置，后收编

风险四：

- 为了体验统一，前端不断侵入首页和设置页

应对：

- 第一阶段坚持独立页面策略

## 20. 最终建议

当前最合适的路线不是“大改现有主流程”，也不是“现在就仓外拆服务”，而是：

- 在同一仓库内新增一个 AI 选股独立业务域
- 按分层方式组织目录和职责
- 通过少量挂载点和现有系统集成
- 先做可回测、可解释、可跟踪的 MVP
- 功能稳定后，再逐步接入调度、通知、组合与更深的 AI 复核

这条路线的关键不在于“目录分开”本身，而在于：

- 业务域独立
- 数据语义独立
- 配置边界独立
- 对主仓的侵入最小

如果后续继续按这个约束演进，AI 选股模块既能在当前仓库内高效落地，也能尽可能保留持续跟进上游的能力。
