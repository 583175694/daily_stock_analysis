## 4. 背景与设计目标

### 4.1 背景

当前项目已经具备：

- 多市场行情与信息源接入
- 单股分析、大盘复盘、Agent 问股、回测、组合管理
- Web、桌面端、Bot、API、通知、定时任务等基础能力

最自然的能力延伸，不是继续强化“分析一只已经给出的股票”，而是增加“先从股票池里筛出值得关注的候选，再解释、跟踪和验证”的 AI 选股能力。

### 4.2 设计目标

- 在同仓库内新增独立的 AI 选股业务域
- 最大化复用现有数据、分析、通知、回测、Web 与 Agent 能力
- 最小化对现有单股分析主流程和高频变动文件的侵入
- 支持从 V1 到 V3 的渐进式演进

### 4.3 非目标

- 第一阶段不把 AI 选股做成仓外独立服务
- 不重写现有单股分析、历史记录、回测或组合管理主链路
- 不把选股结果强塞进现有分析历史表的既有语义
- 不为“统一体验”大范围重构首页、设置页和主调度入口

## 5. 关键约束与设计原则

### 5.1 关键约束

- 代码仍在同一仓库内开发
- 优先做加法式扩展，而不是改写现有模块
- 上游后续仍可能持续变更 API、Web、Agent、配置、数据源与调度逻辑
- 新能力未来可能继续扩展，因此现在的边界需要考虑后续维护成本

### 5.2 设计原则

- 业务域独立：AI 选股是新业务域，不是单股分析的附属步骤
- 薄集成：只在少量入口和适配层与现有系统交汇
- 分层复用：复用现有公共能力，但不把依赖散落到核心主流程
- 数据隔离：新表、新任务、新配置命名空间，避免污染现有语义
- 渐进演进：先做好 V1，再叠加 V2、V3
- 先内置后开放：先模板化，再参数化，最后自然语言输入

## 6. 方案结论

结论：

- 采用“同仓独立业务域 + 薄集成挂载”

不建议：

- 把选股逻辑直接揉进现有单股分析主流程
- 一开始就引入通用插件平台
- 当前阶段拆成仓外独立服务

原因：

- 同仓复用效率高
- 冲突面可控
- 业务边界清晰
- 后续功能继续扩展时，维护成本更低

## 7. 推荐架构

### 7.1 总体结构

建议在 `src/` 下维护独立业务域，例如：

```text
src/stock_picker/
  domain/
  application/
  infrastructure/
  interfaces/
```

### 7.2 推荐职责划分

- `domain/`
  - 股票池定义
  - 初筛规则
  - 因子与评分模型
  - 候选股实体
  - 排名与入选逻辑
  - 跟踪结果与验证规则
- `application/`
  - 运行选股任务
  - 读取股票池
  - 生成候选
  - 调用 AI 做解释与复核
  - 保存结果、生成摘要、触发通知
  - 跟踪历史表现
- `infrastructure/`
  - 数据源适配
  - 分析能力适配
  - LLM / Agent 适配
  - 通知适配
  - 数据库存储实现
  - 回测与组合能力适配
- `interfaces/`
  - API 暴露
  - 定时任务入口
  - 可选 CLI 入口

## 8. 与现有仓库的薄集成点

建议只在少数位置挂载：

- 新增 API 路由注册
- 新增 Web 页面路由和侧边栏入口
- 新增选股运行入口
- 可选增加定时任务挂点

适合保持薄改动的区域：

- `api/v1/router.py`
- Web 路由入口
- 侧边栏导航
- 少量调度挂载文件

## 9. 明确不建议深碰的区域

以下区域除非有明确必要，不建议作为主改动面：

- 现有单股分析核心流水线
- 现有分析报告主结构与历史记录主语义
- 数据源总入口和 fallback 主逻辑
- 全局配置主模型的大规模字段扩张
- 首页主看板的大面积重构
- 设置页的大规模侵入式改造
- `main.py` 中持续增长的既有启动参数和主调度分支

原因：

- 这些区域是高冲突风险区
- 深度侵入后，后续同步和维护成本会明显上升

## 10. 版本边界与已拍板结论

## 11. 模块边界设计

### 11.1 板块层

职责：

- 提供行业、概念、主题等板块维度的候选范围
- 支持用户先选板块，再进入板块内个股筛选
- 在扩大选股范围的同时，避免直接进入全量市场扫描

V2 结论：

- 优先支持“用户手动选板块 -> 板块内选股”
- 板块筛选优先使用结构化指标，不让 LLM 直接对全量板块做自由判断

V3 扩展：

- 支持“结构化规则收敛候选板块 -> AI 解释/排序候选板块 -> 板块内选股”
- AI 选板块不直接替代板块层结构化筛选，只在候选板块范围内工作

设计原则：

- 板块口径要稳定，避免多套分类混用导致结果不可比较
- 板块层只负责收敛候选范围，不直接替代个股评分层
- 板块层的引入应优先提升覆盖范围和效率，而不是显著增加 token 消耗

### 11.2 股票池层

职责：

- 定义本次执行的股票范围
- 提供股票池元信息
- 对接已有股票列表来源

V1 结论：

- 先支持 `STOCK_LIST`
- 可选支持内置示例池
- 不做用户自定义复杂编辑器

V2 扩展：

- 支持“板块对应股票集”作为股票范围来源之一
- 支持在自选股模式和板块模式之间切换

### 11.3 初筛层

职责：

- 基于确定性规则做硬过滤
- 输出过滤结果和过滤原因

建议过滤项：

- 无法获取基础行情数据的标的过滤
- 最近交易数据缺失过滤
- 日均成交额过低过滤
- 价格异常过滤
- 极端波动过滤
- 不满足模板最低趋势条件过滤

### 11.4 评分层

职责：

- 基于结构化规则打分
- 决定候选排序

V1 建议维度：

- 趋势分
- 量价分
- 板块分
- 新闻分
- 风险扣分

V2 补充建议：

- 板块模式下，先做板块层结构化收敛，再做个股结构化评分
- 不让 LLM 同时承担“选板块 + 选个股”的双层决策

### 11.5 AI 复核与解释层

职责：

- 对入围候选给出解释摘要
- 做语言层复核，不接管排序逻辑

AI 输入建议：

- 股票基本信息
- 各维度评分
- 最近关键行情摘要
- 新闻摘要
- 模板说明

### 11.6 跟踪与验证层

职责：

- 跟踪候选后续表现
- 统计模板效果
- 为 V2 / V3 提供验证闭环

说明：

- 该层主要属于 V2 及以后

## 12. 数据与配置设计

### 12.1 数据存储建议

V1 建议拆分为三类核心数据：

- `picker_tasks`
- `picker_candidates`
- `picker_candidate_scores`

目的：

- 区分任务、候选结果、评分明细
- 支持回看、解释和后续统计

V2 补充建议：

- 若启用板块优先选股，需记录任务所使用的板块范围

V3 补充建议：

- 若启用 AI 选板块，建议保留候选板块列表、板块摘要和最终推荐结果，方便后续复盘

### 12.2 配置设计建议

第一阶段：

- 模板元数据写在代码中
- 允许按市场覆盖部分阈值
- 不做复杂可视化配置

第二阶段：

- 支持 `template_id + template_overrides`
- 将参数范围限制在受控字段内

第三阶段：

- 自然语言输入先归一化为 `template_id + template_overrides`
- 不允许自然语言直接进入执行层

## 17. 风险与控制

主要风险：

- 板块口径不稳定，导致结果不可比较
- 板块层和个股层都交给 AI，造成 token 成本和结果波动同时上升
- 业务边界再次被拉回主分析流程
- 模板与评分策略在 V2 提前失控
- 自然语言输入过早进入执行层，导致不可解释与不可回放
- 统计口径不一致，影响 V2 验证结果可信度

控制原则：

- 保持独立业务域
- 保持结构化评分为主、AI 解释为辅
- 板块层优先用结构化规则收敛候选范围
- 所有新增能力按 `V1 -> V2 -> V3` 顺序推进
- 新能力先收口产品边界，再落地实现

## 19. V3 之后的专业化演进蓝图

说明：

- 本章节不做具体排期
- 本章节聚焦能力层设计，而不是下一次迭代的开发清单
- 本章节的目标是回答“若继续做深，技术与平台上应如何从 MVP 走向更专业的投研体系”

### 19.1 对现状的技术判断

- 当前实现已经具备独立业务域、任务沉淀、候选沉淀、后验沉淀与统计入口
- 当前实现适合作为“可运行、可回看、可验证”的第一阶段底座
- 当前实现仍偏轻量化：
  - 板块口径依赖现有股票清单字段
  - 评估窗口固定
  - 模板统计维度较少
  - 组合与执行尚未进入体系
  - AI 仍主要承担解释而不是研究助手角色

结论：

- 架构方向可继续沿用
- 但数据层、策略层、评估层和平台层需要明显升级，才能进入更专业的长期形态

### 19.2 建议的能力层次

#### 1. 数据标准化层

建议补强：

- 正式行业分类映射表
- 概念题材映射表
- 风格标签与市值/流动性分层
- 财务因子表
- 资金因子表
- 事件因子表

### 19.3 需要的数据源方向

优先级较高的数据源类型：

- A 股正式财务与估值数据
- 行业与概念分类数据
- 公告与事件数据
- 资金流与席位数据
- 研报与新闻摘要数据
- 指数、行业指数、风格指数数据

可选的平台或能力方向：

- Tushare Pro / AkShare
- 交易所公告与巨潮资讯抓取
- 行业分类授权数据
- 研报聚合或新闻聚合能力
- 券商 API / 模拟交易平台

说明：

- 具体供应商选择不应先于数据字典与字段标准化设计
- 先定义统一字段，再接数据源，能显著降低后续迁移成本

### 19.4 需要的平台能力

- 特征仓库
- 统一评估任务调度器
- 模板实验与对照平台
- 数据质量监控
- Prompt 版本治理
- 成本监控
- 审计与复现能力
- 研究结论的结构化落库

### 19.5 未来设计原则

- 优先提升研究质量，而不是先增加更多入口
- 优先提升评估与归因质量，而不是先扩大 AI 判断范围
- 优先让结果可验证、可比较、可复盘，而不是先追求“更聪明的单次推荐”
- 若某能力无法形成稳定口径和可复现实验，则不应直接进入主链路
