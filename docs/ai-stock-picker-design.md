## 4. 背景与设计目标

### 4.1 背景

当前项目已经具备：

- 多市场行情与信息源接入
- 单股分析、大盘复盘、Agent 问股、回测、组合管理
- Web、桌面端、Bot、API、通知、定时任务等基础能力

最自然的能力延伸，不是继续强化“分析一只已经给出的股票”，而是增加“先从股票池里筛出值得关注的候选，再解释、跟踪和验证”的 AI 选股能力。

### 4.2 设计目标

- 在同仓库内新增独立的 AI 选股业务域
- 最大化复用现有数据、分析、通知、回测、Web 与 Agent 能力
- 最小化对现有单股分析主流程和高频变动文件的侵入
- 支持从 V1 到 V6 的渐进式演进

### 4.3 非目标

- 第一阶段不把 AI 选股做成仓外独立服务
- 不重写现有单股分析、历史记录、回测或组合管理主链路
- 不把选股结果强塞进现有分析历史表的既有语义
- 不为“统一体验”大范围重构首页、设置页和主调度入口

## 5. 关键约束与设计原则

### 5.1 关键约束

- 代码仍在同一仓库内开发
- 优先做加法式扩展，而不是改写现有模块
- 上游后续仍可能持续变更 API、Web、Agent、配置、数据源与调度逻辑
- 新能力未来可能继续扩展，因此现在的边界需要考虑后续维护成本

### 5.2 设计原则

- 业务域独立：AI 选股是新业务域，不是单股分析的附属步骤
- 薄集成：只在少量入口和适配层与现有系统交汇
- 分层复用：复用现有公共能力，但不把依赖散落到核心主流程
- 数据隔离：新表、新任务、新配置命名空间，避免污染现有语义
- 渐进演进：先做好 V1、V2，再补强 V3 核心能力，完成 V4 精度增强后，再扩展到 V5
- 先内置后开放：先模板化，再参数化，最后自然语言输入

## 6. 方案结论

结论：

- 采用“同仓独立业务域 + 薄集成挂载”

不建议：

- 把选股逻辑直接揉进现有单股分析主流程
- 一开始就引入通用插件平台
- 当前阶段拆成仓外独立服务

原因：

- 同仓复用效率高
- 冲突面可控
- 业务边界清晰
- 后续功能继续扩展时，维护成本更低

## 7. 推荐架构

### 7.1 总体结构

建议在 `src/` 下维护独立业务域，例如：

```text
src/stock_picker/
  domain/
  application/
  infrastructure/
  interfaces/
```

### 7.2 推荐职责划分

- `domain/`
  - 股票池定义
  - 初筛规则
  - 因子与评分模型
  - 候选股实体
  - 排名与入选逻辑
  - 跟踪结果与验证规则
- `application/`
  - 运行选股任务
  - 读取股票池
  - 生成候选
  - 调用 AI 做解释与复核
  - 保存结果、生成摘要、触发通知
  - 跟踪历史表现
- `infrastructure/`
  - 数据源适配
  - 分析能力适配
  - LLM / Agent 适配
  - 通知适配
  - 数据库存储实现
  - 回测与组合能力适配
- `interfaces/`
  - API 暴露
  - 定时任务入口
  - 可选 CLI 入口

## 9. 明确不建议深碰的区域

以下区域除非有明确必要，不建议作为主改动面：

- 现有单股分析核心流水线
- 现有分析报告主结构与历史记录主语义
- 数据源总入口和 fallback 主逻辑
- 全局配置主模型的大规模字段扩张
- 首页主看板的大面积重构
- 设置页的大规模侵入式改造
- `main.py` 中持续增长的既有启动参数和主调度分支

原因：

- 这些区域是高冲突风险区
- 深度侵入后，后续同步和维护成本会明显上升

## 10. 版本边界与已拍板结论

### 10.1 V1 已拍板结论

- V1 只做内置模板
- V1 不支持自然语言自由输入策略
- V1 排序由结构化规则与结构化评分决定
- V1 的 AI 只负责解释和复核，不负责最终排序
- V1 股票池至少支持 `STOCK_LIST`
- V1 可支持内置预设池，但不做复杂股票池编辑器
- V1 允许三市场共存，但模板可声明适用市场
- V1 允许数据缺失时按 fail-open 方式降级部分评分
- V1 聚焦手动运行，不做定时任务

### 10.2 V2 待拍板项

- 板块数据来源与分类口径
- 板块模式是否只支持用户手动选板块
- 参数化模板开放哪些字段
- 候选后验跟踪的时间窗口
- 模板效果统计默认指标
- 通知摘要的渠道与频率

### 10.3 V3 已拍板方向

- V3 先做核心完善，不在该阶段新增放大器型入口
- V3 的验收重点是口径稳定、模板质量、解释一致性与后验补验
- 板块层增强服务于后续 `AI 选板块`，但不在 V3 直接引入全量 AI 板块推荐

### 10.4 V4 已拍板方向

- V4 先做精度增强，不在该阶段继续扩自动化和自由输入入口
- V4 的验收重点是交易规则完整、分层统计可用、市场环境过滤有效、真实约束可回放
- AI 在 V4 优先承担复核、风险提示和置信度表达，不直接替代结构化排序
- `高置信度` 不是文案标签，而是校准后结论；未通过样本外验证、滚动验证与分桶校准前，不应对外承诺高置信度
- V4 默认把置信度拆为 `研究置信度` 与 `执行置信度` 两层，避免“逻辑成立”与“交易可做”混成单一分数

### 10.5 V5 待拍板项

- 关键行业 / 概念 / 事件 / 宏观标签的最小闭环范围
- 公开免费源与有限额源分别覆盖哪些字段，哪些字段允许降级
- 模板版本化、策略家族化与因子贡献归因的最小闭环定义
- 多基准 / 多环境评估首版默认覆盖哪些基准与环境切面
- AI 研究副驾输出哪些固定结构，如何与结构化评分结果对齐

### 10.6 V6 待拍板项

- AI 选板块的候选板块生成规则
- AI 选板块默认返回多少个候选板块
- 自然语言策略输入的可控边界
- 定时任务的执行频率与调度入口
- 组合联动展示的默认视图
- 市场环境动态调节的触发条件

## 11. 模块边界设计

### 11.1 板块层

职责：

- 提供行业、概念、主题等板块维度的候选范围
- 支持用户先选板块，再进入板块内个股筛选
- 在扩大选股范围的同时，避免直接进入全量市场扫描

V2 结论：

- 优先支持“用户手动选板块 -> 板块内选股”
- 板块筛选优先使用结构化指标，不让 LLM 直接对全量板块做自由判断

V6 扩展：

- 支持“结构化规则收敛候选板块 -> AI 解释/排序候选板块 -> 板块内选股”
- AI 选板块不直接替代板块层结构化筛选，只在候选板块范围内工作

设计原则：

- 板块口径要稳定，避免多套分类混用导致结果不可比较
- 板块层只负责收敛候选范围，不直接替代个股评分层
- 板块层的引入应优先提升覆盖范围和效率，而不是显著增加 token 消耗

### 11.2 股票池层

职责：

- 定义本次执行的股票范围
- 提供股票池元信息
- 对接已有股票列表来源

V1 结论：

- 先支持 `STOCK_LIST`
- 可选支持内置示例池
- 不做用户自定义复杂编辑器

V2 扩展：

- 支持“板块对应股票集”作为股票范围来源之一
- 支持在自选股模式和板块模式之间切换

### 11.3 初筛层

职责：

- 基于确定性规则做硬过滤
- 输出过滤结果和过滤原因

建议过滤项：

- 无法获取基础行情数据的标的过滤
- 最近交易数据缺失过滤
- 日均成交额过低过滤
- 价格异常过滤
- 极端波动过滤
- 不满足模板最低趋势条件过滤

### 11.4 评分层

职责：

- 基于结构化规则打分
- 决定候选排序

V1 建议维度：

- 趋势分
- 量价分
- 板块分
- 新闻分
- 风险扣分

V2 补充建议：

- 板块模式下，先做板块层结构化收敛，再做个股结构化评分
- 不让 LLM 同时承担“选板块 + 选个股”的双层决策

### 11.5 AI 复核与解释层

职责：

- 对入围候选给出解释摘要
- 做语言层复核，不接管排序逻辑

AI 输入建议：

- 股票基本信息
- 各维度评分
- 最近关键行情摘要
- 新闻摘要
- 模板说明

### 11.6 跟踪与验证层

职责：

- 跟踪候选后续表现
- 统计模板效果
- 为 V2 / V3 / V4 / V5 提供验证闭环

说明：

- 该层主要属于 V2 及以后

## 17. 风险与控制

主要风险：

- 板块口径不稳定，导致结果不可比较
- 板块层和个股层都交给 AI，造成 token 成本和结果波动同时上升
- 业务边界再次被拉回主分析流程
- 模板与评分策略在 V2 提前失控
- 自然语言输入过早进入执行层，导致不可解释与不可回放
- 统计口径不一致，影响 V2 验证结果可信度

控制原则：

- 保持独立业务域
- 保持结构化评分为主、AI 解释为辅
- 板块层优先用结构化规则收敛候选范围
- 所有新增能力按 `V1 -> V2 -> V3 -> V4 -> V5 -> V6` 顺序推进
- 新能力先收口产品边界，再落地实现

## 19. V6 之后的专业化演进蓝图

### 19.1 对现状的技术判断

#### 19.3.3 当前信息源不足的能力

- 正式行业 / 概念分类体系驱动的板块研究与行业中性比较
- 结构化公告 / 事件驱动因子
- 研报、盈利预期修订与一致预期变化
- 多市场统一的财务、估值、行业、事件与基准口径
- 组合层相关性、风格暴露与边际贡献分析

#### 19.3.4 V4 优先补充的外部信息源

- 正式行业分类数据：用于行业中性比较、板块轮动、龙头识别与组合暴露分析
- 结构化公告 / 事件数据：用于业绩预告、回购增持、减持、解禁、问询、停复牌等事件因子
- 指数、行业指数、风格指数与多基准序列：用于市场环境识别、多基准评估与风格相对强弱
- 研报 / 新闻摘要与盈利预期修订数据：用于 AI 研究副驾、多空论点、反例挑战与模板失效诊断

#### 19.3.5 `V4.1` 与 `V4.2` 的数据源策略

- `V4.1` 优先复用现有 `Tushare Pro / AkShare / efinance / 搜索服务` 能力，只在必要处补少量外部数据
- `V4.2` 默认需要新增外部信息源；若外部数据尚未到位，应优先缩小能力范围，而不是降低口径要求
- 多市场专业化默认遵循 `A 股先做深 -> 港股 / 美股再补齐口径` 的顺序推进
- 供应商选型不应先于数据字典、字段标准化与可回放落库结构设计

当前阶段已决定的 A 股范围约束：

- 当前只做 A 股深度优化，暂不为港股、美股补同等级别的数据源链路
- 已知仓库持有可用 `Tushare Pro` 积分档时，应优先把 `Tushare Pro` 视为 A 股结构化数据主干之一，而不是仅作为兜底源
- 本阶段默认不引入 `Wind / iFinD / Choice / JQData / RQData` 等销售报价型数据源，先把公开免费源与免费但有限额 / 有限权的数据源接入价值榨干

#### 19.3.6 已纳入方案的公开免费源与有限额 / 有限权源

优先纳入的公开免费源：

- `巨潮资讯 / cninfo`
  - 角色：A 股法定公告原始源
  - 用途：公告原文、财报披露、临时公告、事件抽取底座
  - 当前状态：仓库尚无独立 `cninfo` 适配器；现有链路仅在少量场景通过搜索或 `AkShare` 间接触达，后续需补独立接入与结构化抽取
- `中证指数 / csindex`
  - 角色：行业指数、风格指数、多基准方法论来源
  - 用途：市场环境识别、行业/风格相对强弱、多基准评估
  - 当前状态：仓库尚无独立指数数据拉取适配器，后续需新增指数序列与分类映射接入
- `国证指数 / cnindex`
  - 角色：行业分类与行业指数补充源
  - 用途：行业归属、行业轮动、行业中性比较、估值分位补充
  - 当前状态：仓库尚无独立适配器，后续需新增分类结果与指数序列接入
- `国家统计局 / 人民银行 / Shibor / 中国货币网`
  - 角色：宏观与流动性环境原始源
  - 用途：市场环境标签、利率与信用环境、宏观 regime 辅助识别
  - 当前状态：仓库尚无独立宏观数据接入，后续需新增离线抓取 / 定时同步链路

优先纳入的免费但有限额 / 有限权源：

- `Tushare Pro`
  - 角色：A 股结构化主干数据源之一
  - 用途：日线、实时行情补充、资金流、财务指标、分红、龙虎榜、申万 / 中信行业分类、部分研报与公告相关接口
  - 当前状态：已集成，可直接通过 `TUSHARE_TOKEN` 配置；在已具备 `2000` 积分档时，应优先用于 `V4` 的实时补充、结构化财务与分类字段增强
- `AkShare`
  - 角色：公开网页聚合与研究验证补充源
  - 用途：东方财富 / 新浪 / 腾讯等公开站点的行情、公告、宏观、龙虎榜、板块等补充数据
  - 当前状态：已集成，无需单独 Key；更适合研究验证与公开信息补洞，不作为唯一生产真源
- `efinance`
  - 角色：东方财富实时与板块统计补充源
  - 用途：A 股实时行情、指数行情、市场涨跌统计、板块排行
  - 当前状态：已集成，无需单独 Key；适合作为实时与市场统计增强源
- `pytdx`
  - 角色：通达信行情补充源
  - 用途：日线、分钟线、部分实时链路兜底
  - 当前状态：已集成，无需 Key；可通过通达信服务器或自定义 `PYTDX_HOST / PYTDX_PORT` 使用
- `Baostock`
  - 角色：基础历史行情与部分财务补充源
  - 用途：历史日线、基础财务字段与低成本回补
  - 当前状态：已集成，无需 Key；适合作为低频补充源

说明：

- `公开免费源` 主要解决“官方原始披露、行业 / 风格基准、宏观环境”问题
- `免费但有限额 / 有限权源` 主要解决“结构化日常研究数据、实时行情、基础财务与板块统计”问题
- `V4.1` 默认先把以上两类源用好，再决定是否需要更重的数据采购

#### 19.3.8 当前仍需额外集成的数据源

虽然已经决定纳入方案，但当前仓库还没有可直接配置的独立接入层：

- `巨潮资讯 / cninfo`
  - 缺口：缺少公告检索、公告正文抓取、公告元数据标准化、事件抽取落库
- `中证指数 / csindex`
  - 缺口：缺少行业 / 风格 / 多基准指数序列拉取与本地缓存
- `国证指数 / cnindex`
  - 缺口：缺少行业分类结果同步、行业指数序列与分类映射
- `国家统计局 / 人民银行 / Shibor / 中国货币网`
  - 缺口：缺少宏观数据抓取、市场环境标签生成与定时更新链路

当前阶段明确不纳入本轮接入范围：

- `Wind`
- `iFinD`
- `Choice`
- `JQData`
- `RQData`

优先级较高的数据源类型：

- A 股正式财务与估值数据
- 行业与概念分类数据
- 公告与事件数据
- 资金流与席位数据
- 研报与新闻摘要数据
- 指数、行业指数、风格指数数据

可选的平台或能力方向：

- Tushare Pro / AkShare
- 交易所公告与巨潮资讯抓取
- 行业分类授权数据
- 研报聚合或新闻聚合能力
- 券商 API / 模拟交易平台

说明：

- 具体供应商选择不应先于数据字典与字段标准化设计
- 先定义统一字段，再接数据源，能显著降低后续迁移成本

### 19.4 需要的平台能力

- 特征仓库
- 统一评估任务调度器
- 模板实验与对照平台
- 数据质量监控
- Prompt 版本治理
- 成本监控
- 审计与复现能力
- 研究结论的结构化落库

### 19.5 未来设计原则

- 优先提升研究质量，而不是先增加更多入口
- 优先提升评估与归因质量，而不是先扩大 AI 判断范围
- 优先让结果可验证、可比较、可复盘，而不是先追求“更聪明的单次推荐”
- 若某能力无法形成稳定口径和可复现实验，则不应直接进入主链路
