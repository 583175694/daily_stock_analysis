# AI 选股开发前确认清单

## 1. 文档定位

本文档是 AI 选股 V1 开发前的最后一层收口清单。

它不再讨论：

- 为什么要做
- 总体架构怎么分层
- 长期版本规划怎么演进

它只回答一件事：

- 在正式开始编码前，哪些关键问题必须先拍板

相关文档：

- `docs/ai-stock-picker-prd.md`
- `docs/ai-stock-picker-design.md`
- `docs/ai-stock-picker-implementation-plan.md`

## 2. 使用方式

建议把本清单作为开发启动前的评审单。

规则：

- `必须确认` 项全部有结论，才能进入编码
- `建议确认` 项可以边开发边细化，但最好提前达成一致
- `暂缓处理` 项本轮不要扩 scope

## 3. V1 范围锁定

### 3.1 必须确认：V1 功能范围

V1 建议只保留以下功能：

- 股票池选择
- 模板选择
- 手动运行选股
- 候选结果列表
- 候选详情
- 一键发起单股分析
- 一键跳转 Agent 问股
- 一键加入自选股
- 历史任务回看

### 3.2 必须确认：V1 不做的功能

V1 建议明确排除：

- 定时自动选股
- 自动通知
- 后验跟踪
- 模板效果统计
- 组合联动
- 用户自定义股票池编辑器
- 全市场全量扫描

确认结论：

- 是否接受上述边界
- 是否还有必须放进 V1 的功能

## 4. 模板确认

### 4.1 必须确认：V1 模板数量

建议 V1 只做 3 到 5 个模板。

推荐默认值：

- 先做 3 个，减少实现复杂度

建议首批模板：

- `trend_breakout`
- `strong_pullback`
- `balanced`

可选扩展模板：

- `sector_leader`
- `event_driven`

### 4.2 必须确认：模板定义方式

必须拍板的问题：

- 模板是写死在代码里，还是采用配置驱动
- V1 是否允许不同市场复用同一模板
- 模板是否允许不同市场有不同阈值
- V1 是否只允许内置模板
- V1 是否明确不支持自然语言自由输入策略

推荐方案：

- V1 模板元数据写在代码中
- V1 先允许跨市场复用
- 阈值可在模板配置中按市场覆盖，但不做复杂可视化编辑
- V1 只允许内置模板
- V1 不支持自然语言自由输入

### 4.2.1 必须确认：后续策略输入的演进路径

建议明确三阶段策略输入模式：

- V1：内置模板
- V2：参数化模板
- V3：自然语言策略输入

必须拍板的问题：

- 是否接受上述节奏
- 是否有人坚持把自然语言输入提前到 V1

推荐方案：

- 不提前
- V1 先把模板体系做稳，再开放外部输入

### 4.3 建议确认：模板解释文案

每个模板建议提前确认：

- 对用户显示的中文名称
- 一句话说明
- 适用市场
- 风险提示

## 5. 股票池确认

### 5.1 必须确认：V1 股票池来源

建议 V1 只支持两类：

- 当前 `STOCK_LIST`
- 内置预设池

必须拍板的问题：

- 是否只先支持 `STOCK_LIST`
- 内置预设池是否要在 V1 就提供
- 预设池如果提供，具体包含哪些池

推荐方案：

- V1 至少支持 `STOCK_LIST`
- 若预设池工作量可控，可补一个“示例池”机制

### 5.2 建议确认：市场范围

必须明确：

- V1 是否支持 A 股、港股、美股同时运行
- 是否允许模板只在某些市场可用

推荐方案：

- V1 支持三市场
- 模板可声明适用市场，但前端先统一展示

### 5.3 暂缓处理

本轮建议不做：

- 板块池编辑器
- 指数成分同步器
- 用户上传股票池

## 6. 数据需求确认

### 6.1 必须确认：V1 评分依赖哪些数据

建议先只依赖这些数据：

- 最近一段时间日线行情
- 成交量或成交额
- 板块信息
- 基础新闻摘要

可选但不作为 V1 强依赖：

- 复杂基本面聚合
- 社交情绪
- 深度公告解析

必须拍板的问题：

- 若新闻源不可用，是否允许降级继续出结果
- 若板块信息缺失，是否允许只做部分评分

推荐方案：

- 都允许 fail-open 降级

### 6.2 必须确认：AI 解释依赖什么输入

建议 AI 输入只包含：

- 股票基本信息
- 各维度评分
- 最近关键行情摘要
- 新闻摘要
- 模板说明

建议不要直接把全量原始数据丢给 AI。

## 7. 评分机制确认

### 7.1 必须确认：V1 评分维度

建议固定为：

- 趋势分
- 量价分
- 板块分
- 新闻分
- 风险扣分

### 7.2 必须确认：是否允许 LLM 决定排序

推荐结论：

- 不允许

原因：

- V1 需要可解释、可测试、可回看
- 排序必须由结构化规则和结构化打分完成
- AI 只做解释和复核

### 7.2.1 必须确认：是否允许 LLM 自由定义策略

推荐结论：

- 不允许

原因：

- V1 的 AI 作用应是解释模板结果，而不是自由创建不可控的筛选逻辑
- 否则会明显增加回测、前端展示和错误兜底复杂度

### 7.3 建议确认：Top N 数量

建议确认：

- 默认返回多少只候选
- AI 解释前多少只候选

推荐默认值：

- 列表显示 Top 20
- AI 解释 Top 10

## 8. 任务模型确认

### 8.1 必须确认：V1 是同步还是异步

两种方案：

- 同步执行后直接返回结果
- 语义按异步任务设计，由前端轮询任务详情

推荐方案：

- 接口语义按异步任务设计
- 实现上可先简化，但前端和数据结构不要写死成同步语义

### 8.2 必须确认：历史任务保留什么信息

建议至少保存：

- 模板
- 股票池
- 候选结果
- 评分明细
- AI 解释
- 执行时间
- 失败原因

## 9. API 确认

### 9.1 必须确认：V1 API 清单

建议固定为：

- `GET /api/v1/picker/templates`
- `GET /api/v1/picker/universes`
- `POST /api/v1/picker/run`
- `GET /api/v1/picker/tasks`
- `GET /api/v1/picker/tasks/{task_id}`

### 9.2 必须确认：返回结构风格

必须拍板：

- 是否遵循现有 API 成功/失败风格
- 是否保留统一的 `success/error` 语义
- 候选详情是否在任务详情接口中一次返回

推荐方案：

- 尽量贴近现有 API 风格
- 任务详情接口一次返回完整结果，减少前端拼装复杂度

### 9.3 建议确认：前端轮询方式

建议：

- V1 不做 SSE
- 任务结果采用简单列表 + 详情查询

## 10. 数据表确认

### 10.1 必须确认：V1 表数量

建议 V1 最少 3 张表：

- `picker_tasks`
- `picker_candidates`
- `picker_candidate_scores`

### 10.2 必须确认：是否需要额外存原始快照

需要拍板：

- 是否把 AI 输入摘要或数据快照存下来

推荐方案：

- 只存必要摘要和评分细节
- V1 不强求保存所有原始行情和新闻全文

### 10.3 建议确认：唯一键与幂等

建议明确：

- 同一模板 + 同一股票池 + 同一天是否允许重复跑多次
- 历史任务是否允许重复结果并存

推荐方案：

- 允许重复运行
- 每次都作为独立任务保存

## 11. 页面交互确认

### 11.1 必须确认：入口位置

建议：

- 侧边栏新增独立入口 `AI 选股`

### 11.2 必须确认：详情展示方式

二选一：

- 抽屉
- 分栏详情

推荐方案：

- 抽屉

原因：

- 改动小
- 更适合复用现有页面布局

### 11.3 必须确认：候选结果的核心操作

建议固定为：

- 查看详情
- 发起单股分析
- 跳转 Agent 问股
- 加入自选股

### 11.4 建议确认：历史任务入口

建议：

- 同页内展示最近任务列表
- 先不单独开二级页面

## 12. 与现有系统的边界确认

### 12.1 必须确认：哪些区域不碰

建议写死为本轮开发纪律：

- 不改单股分析主流程
- 不改现有分析历史主语义
- 不改现有回测主链路
- 不大改首页
- 不大改设置页
- 不在 `main.py` 中新增复杂调度逻辑

### 12.2 必须确认：哪些区域允许薄改动

建议允许：

- API 路由注册
- 前端页面路由注册
- 侧边栏导航入口
- 少量公共类型和样式复用

## 13. 测试与验收确认

### 13.1 必须确认：V1 最低验证范围

后端最低验证建议：

- 新增模块基础导入无报错
- 关键 Python 文件通过 `py_compile`
- 至少覆盖一次手动运行主链路

前端最低验证建议：

- 页面路由可打开
- 可发起任务
- 可展示结果

### 13.2 建议确认：首版验收口径

建议用下面 5 条判断是否可以合并：

- 页面可访问
- 可手动运行一次选股
- 能稳定返回候选列表
- 每只候选有解释
- 可联动单股分析

## 14. 推荐默认拍板结果

如果希望尽快进入开发，建议直接采用下面这组默认值：

- V1 只做手动运行，不做定时和通知
- 股票池先只支持 `STOCK_LIST`
- 模板先做 3 个：`trend_breakout`、`strong_pullback`、`balanced`
- 排序只靠结构化评分，AI 不参与排序
- 策略来源只允许内置模板，不开放自然语言自由输入
- 默认返回 Top 20，AI 解释 Top 10
- 页面采用独立 `/picker` 路由
- 详情采用抽屉形式
- 后端新增 3 张表
- 接口采用 `/api/v1/picker/*`
- 不碰单股分析主流程和主调度逻辑

## 15. 开工门槛

只有当以下问题全部有明确答案，才建议正式进入编码：

- V1 范围是否锁定
- 首批模板是否锁定
- 股票池来源是否锁定
- 策略来源是否锁定为“仅内置模板”
- API 清单是否锁定
- 数据表是否锁定
- 页面入口与详情交互是否锁定
- 不碰区域是否锁定

若以上仍有明显分歧，建议继续先做方案收口，不要直接开工。
